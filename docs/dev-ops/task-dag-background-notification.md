# Task DAG and Background Tool Notifications

## Goal

The CodeOps agent now records the execution loop as task-level runtime structures instead of only as a flat step list.
This keeps the current synchronous service contract intact while making the harness ready for long-running tools,
multi-agent scheduling, and UI progress views.

## Runtime Structures

- `taskDagNodes`: ordered DAG nodes for each selected skill. Each node records `nodeId`, `stepNo`, `skillId`,
  `stage`, `status`, `owner`, `blockedBy`, `summary`, and runtime artifacts.
- `backgroundToolTasks`: slow tool execution records. Maven verification is the first integrated tool and records
  request summary, command, status, result summary, abbreviated output, and cost. When real test execution is disabled,
  Maven plans are recorded as `SKIPPED` background tasks instead of silently disappearing.
- `taskNotifications`: event feed generated by background tool tasks. Maven verification emits started and finished
  notifications. Terminal notifications carry a consumed marker so the agent harness can process each result once.
- `consumedTaskNotifications`: compact snapshots of terminal notifications consumed by the loop or task agent. This is
  the bridge from background execution back into model-visible observations and working memory.

The first implementation stores these structures in `EngineeringTaskEntity.context` so the API can expose them without
schema changes. They can be promoted to dedicated tables when task persistence moves beyond the in-memory task store.

## Current Flow

1. `EngineeringTaskAgentService` marks a DAG node as `RUNNING` before a skill executes.
2. The skill runs through the existing agent runtime and permission-governed tool gateway.
3. Maven verification calls go through `BackgroundToolTaskService`, which records background task state and
   notifications around the existing `repo.maven` execution. Disabled execution produces a skipped notification.
4. Before each decision turn, the agent loop consumes terminal background notifications and writes them into
   `backgroundTaskObservations`.
5. Async test verification returns `WAITING_BACKGROUND_TASK` while Maven is still running. The task is parked as
   `WAITING_BACKGROUND_TASK` instead of moving on to release risk with incomplete evidence.
6. When a test verification notification is consumed, success updates test memory and lets the orchestrator continue;
   failure is converted into the existing reflection failure path.
7. The agent loop marks the DAG node as `COMPLETED`, `FAILED`, or `BLOCKED`.
8. `EngineeringTaskTraceService` exposes DAG nodes, background tasks, and notifications in working memory summary,
   step highlights, and test verification stage artifacts.

## Async Tool Mode

The tool registry now exposes a real async command path:

- `repo.maven_background`: starts Maven in a background worker and returns `backgroundTaskId` immediately.
- `task.background_status`: reads the latest status for a background task id.
- `BackgroundToolTaskService`: updates task context and emits `BACKGROUND_TASK_STARTED`,
  `BACKGROUND_TASK_FINISHED`, or `BACKGROUND_TASK_FAILED` notifications from the worker.
- `TestVerificationSkill`: keeps synchronous behavior by default, but `context.asyncTestExecution=true` queues Maven
  verification in the background and returns the queued task id in test execution results with status
  `WAITING_BACKGROUND_TASK`.
- `AgentLoopService`: consumes terminal notifications before asking the model for the next decision and injects
  `backgroundTaskObservations` into loop metadata and trace.
- `EngineeringTaskAgentService`: consumes terminal notifications before orchestrator decisions, parks running async
  verification, and routes failed background test verification into the reflection loop.

## Next Extension

The next natural extension is persistence and resume:

- persist DAG nodes and background task events for recovery across process restarts;
- add cancellation and timeout controls for long-running background tasks.

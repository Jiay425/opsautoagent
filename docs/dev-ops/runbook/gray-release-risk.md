# 灰度发布风险与上线观察手册
## keywords

gray release, canary, rollback, feature flag, release risk, deployment, smoke test, error budget, approval gate

## 适用场景

Agent 已生成代码补丁或配置修复，需要判断是否可以上线、如何灰度、观察哪些指标、什么条件回滚。该手册不用于直接判断事故根因，而是用于发布风险分析和人工审批。

企业级自动修复不能无人值守直接上线。即使编译和测试通过，也要根据改动范围、事故级别、依赖关系、数据一致性风险决定是否人工审批。

## 精确定位片段：代码改动风险分级

单方法空值校验、错误码修正、日志增强通常是低到中风险，但如果触及支付、库存、幂等、事务、权限、缓存一致性，就必须升为高风险。跨多个文件、跨多个方法、修改状态机、修改 SQL、修改序列化字段，都需要人工审批。

如果 patch 修改了 public API、数据库字段、MQ 消息结构、定时任务或配置默认值，即使测试通过，也要列为发布风险点。Agent 不能只根据测试通过判断安全。

## 精确定位片段：灰度策略

高风险修复应按实例比例、租户、用户分组或流量标签灰度。第一阶段只放 1 个实例或 1% 流量，观察 10 到 30 分钟；第二阶段扩大到 10% 到 30%；最终全量。每个阶段都要有明确通过条件和回滚条件。

灰度观察必须绑定事故原始指标。例如修复 order-service 5xx，就观察 5xx rate、P95、P99、错误日志、相关 DB/Redis/RPC 指标。不能只看应用存活。

## 精确定位片段：回滚条件

回滚条件要机器可判定：5xx 连续 3 分钟高于基线 2 倍、P99 连续 5 分钟超过阈值、核心错误日志超过 N 条、DB 连接池 pending 持续大于 0、MQ backlog 持续增长。

对数据修复类变更，回滚代码不一定能回滚数据，需要准备补偿脚本和审计查询。支付、库存、权益类变更必须给出人工确认点。

## 临时止血

如果修复只是配置或容量调整，可以先灰度配置，不走代码发布。若事故仍在扩散，应先限流、降级、扩容或回滚最近发布，再考虑代码修复。

## 长期治理

自动修复平台必须保存补丁 diff、测试结果、风险等级、审批记录、灰度计划和上线观察项。每次发布后应自动复盘：修复是否命中根因，是否引入新告警，是否需要沉淀为事故记忆。

## 误判边界

测试通过不等于可上线。灰度通过也不等于根因已永久解决。如果观察窗口太短或流量没有覆盖关键路径，必须保留风险提示。Runbook 只能提供发布建议，不能替代人工审批。

# 数据库死锁与事务边界排查手册
## keywords

deadlock, lock wait timeout, transaction rollback, innodb deadlock, pessimistic lock, optimistic lock, update order, gap lock, repeatable read

## 适用场景

接口出现间歇性 500、事务回滚、更新超时、库存扣减失败，日志包含 Deadlock found、Lock wait timeout exceeded、CannotAcquireLockException、Transaction rolled back。SkyWalking 中 DB span 可能显示 UPDATE 或 SELECT FOR UPDATE 耗时异常。

死锁不是“数据库偶发问题”这么简单。它往往说明业务更新顺序不一致、事务范围过大、索引缺失导致锁范围扩大，或者高并发下多个资源交叉加锁。

## 精确定位片段：加锁顺序不一致

典型死锁来自两个事务以不同顺序更新资源。例如事务 A 先更新订单再更新库存，事务 B 先更新库存再更新订单。并发时 A 持有订单锁等待库存锁，B 持有库存锁等待订单锁。

修复原则是所有路径统一资源访问顺序。比如下单、取消、补偿、超时关闭都必须按照 `order -> inventory -> account` 或明确约定的顺序执行，不能每个业务方法自己决定顺序。Agent 定位代码时要查所有修改同一批表的方法，而不是只看报错接口。

## 精确定位片段：事务范围过大

如果事务中包含远程 RPC、MQ 发送、文件处理、复杂计算或大量循环更新，锁持有时间会被拉长。慢 SQL 并不一定是单条 SQL 慢，也可能是事务中非数据库操作拖住了已持有的锁。

正确做法是事务内只保留必要的本地一致性写操作。远程调用、消息发送、通知、审计扩展应放到事务外或 outbox。若必须在事务中读取外部结果，必须设置短超时和降级策略。

## 精确定位片段：索引缺失导致锁范围扩大

在 InnoDB Repeatable Read 下，条件更新如果没有命中合适索引，可能扫描大量记录并产生 next-key lock 或 gap lock。表现是业务只想更新一个订单，却锁住一段范围，导致无关请求等待。

需要检查慢 SQL 的 where 条件是否命中索引，尤其是 status、tenant_id、biz_id、created_at 组合条件。对高频状态机更新，推荐使用唯一业务键或主键更新，避免宽范围条件更新。

## 临时止血

短期可以降低并发入口、关闭非核心批处理、拆小批量任务、缩短事务超时。对重复失败的补偿任务要限速，避免补偿任务和在线流量争锁。

如果死锁集中在某个任务，可以暂停该任务并保留在线交易路径。不要盲目提高数据库连接池，因为更多并发可能让锁冲突更严重。

## 长期治理

建立事务规范：禁止事务内远程调用，禁止事务内无限循环，禁止无索引条件更新。对核心表建立加锁顺序文档和代码扫描规则。评测 Harness 应包含并发压测用例，验证高并发下不会出现死锁和状态错乱。

## 误判边界

单次 deadlock 可以由数据库正常检测并回滚，业务若有重试机制不一定是事故。只有死锁频率升高、接口错误率上升、补偿积压或用户可感知失败时，才需要进入事故修复。若只是 Lock wait timeout 且 SQL 本身耗时很长，应先排查慢 SQL 和索引。
